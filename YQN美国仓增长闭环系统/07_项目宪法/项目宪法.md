# YQN 美国仓增长闭环系统项目宪法

## 0. 使用规则

后续任何阶段开始前，必须先读取本文件。若新任务与本文件冲突，以本文件为准；若业务资料不足，输出“待确认”或“灰灯”，不得编造。

## 1. 项目前情提要

项目名称：YQN 美国仓增长闭环系统。

业务背景：用户在 YQN 运去哪负责美国仓 / 北美跨境电商方向的小红书获客增长。当前核心工作不是泛泛行业学习，也不是普通 AI 日报，而是围绕“小红书内容 / 投放 / 私域 / MQL / 销售反馈”建立一个可复盘、可迭代的增长闭环。

用户当前最关心：

1. 小红书带来的 MQL 到底质量怎么样。
2. 哪些素材、选题、渠道更容易带来高质量美国仓客户。
3. 私域、客服、销售、仓库、报价、运营、投放代理分别缺什么信息。
4. 如何把客户痛点反向变成下一轮小红书内容实验。
5. 如何把 YQN 具体业务沉淀成未来 OPC 可复用的 B2B 增长方法论。

重点业务方向：只围绕美国仓 / 北美跨境电商。不得把墨西哥仓作为默认重点。

可参考的 YQN 业务能力包括：美国仓、3PL 仓配、FBA、FBM、SFP、FBA 补货 / FBA Prep、一件代发、退货换标、平台仓转运、B2B 转运、头程、清关、单清、双清、整柜 FCL、拼箱 LCL、自拼箱、尾程派送、OMS / WMS / TMS / BI、多平台发货、库存可视化，以及仓储、出库、尾程、退货等履约成本。任何具体承诺必须有资料证据，没有证据时只能写“待确认”。

系统最终闭环：

素材 / 内容 / 渠道 -> 私域开口 -> MQL 判定 -> MQL 质量复盘 -> 销售反馈 -> 客户痛点沉淀 -> 组织缺口识别 -> 下一轮内容实验 -> 作战日报 -> 再进入素材 / 内容 / 渠道优化。

## 2. 产品定位

这个系统不是：

- 不是普通新闻日报。
- 不是第二大脑炫技项目。
- 不是让用户手动录字段的 MQL 评分器。
- 不是全网爬虫。
- 不是飞书复杂 Bot。
- 不是 CRM / 企微生产系统。

这个系统是：

YQN 美国仓增长闭环系统。第一阶段目标是把用户输入的 MQL 记录、私域记录、销售反馈、投放素材记录、会议纪要、客户原话、产品资料，自动整理成事实提取表、MQL 质量复盘、组织缺口雷达、内容实验库、个人作战日报、管理层摘要和 OPC 通用化模板。

## 3. 角色分工

用户：

- 只负责传递信息。
- 不看代码。
- 不测试代码。
- 不判断产品是否能用。
- 不手动打标。
- 不手动评分。

Codex：

- 负责搭建项目。
- 负责设计样例。
- 负责开发。
- 负责自测。
- 负责提交验收包。
- 负责明确已知问题和下一步建议。

ChatGPT 验收官：

- 负责判断业务方向是否正确。
- 负责判断输出是否有业务价值。
- 负责决定是否进入下一阶段。
- 负责挑战方案，防止系统变成 AI 废话。

## 4. 产品原则

1. 证据优先：所有判断必须有证据片段。没有证据就写“待确认”或“灰灯”，不能编造。
2. 不让用户手动打分：系统可以自动判断红灯、黄灯、绿灯、灰灯，但不能要求用户手动评分。
3. 美国仓优先：所有输出默认围绕美国仓 / 北美跨境电商，不主动展开墨西哥仓。
4. 业务可用优先：输出必须能帮助用户做小红书获客、MQL 质量复盘、组织追问和销售协同，不输出空泛公司宣传。
5. 闭环优先：不是单纯总结信息，而是把信息转成下一轮内容实验和组织动作。
6. 双日报：日报分为个人作战日报和管理层摘要。个人作战日报给用户自己看，管理层摘要给强哥 / Alice 看，必须简洁、业务化、不甩锅。
7. OPC 顺手沉淀：每次输出都要把 YQN 具体内容抽象成未来可复用的 B2B 增长模板。

## 5. 灯号定义

- 绿灯 MQL：证据显示客户属于美国仓 / 北美跨境电商相关场景，且有明确业务量、履约痛点、决策或推进意图。
- 黄灯 MQL：证据显示方向相关且有潜力，但关键字段不足，需要私域、客服或销售补齐。
- 红灯 MQL：证据显示低质量、无效、非目标客户、非业务采购需求，或与美国仓增长目标明显不匹配。
- 灰灯 MQL：证据不足，不能判断。灰灯不是低质量，只是缺少判断依据。

## 6. 阶段 0 验收原则

阶段 0 通过标准：

1. 项目宪法完整。
2. 样例像真实美国仓业务。
3. 每个判断都有证据片段。
4. 红灯和灰灯区分清楚。
5. 内容实验能反向服务小红书获客。
6. 管理层摘要能给老板看。
7. OPC 通用化有复用价值。
8. 没有让用户手动评分。
9. 没有空泛公司宣传。
10. 没有把墨西哥仓作为重点。

## 7. 阶段推进约定

阶段 0 只做闭环样例产品原型，不急着写复杂代码。阶段 1 若进入开发，应先做可重复运行的本地处理流程：读取输入资料、提取事实、生成复盘、生成日报、生成验收包。暂不做复杂飞书 Bot、CRM 写回、全网爬取或自动联系客户。
